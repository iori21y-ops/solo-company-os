## 목적
추측으로 실행을 시작하면 잘못된 전제 위에 작업이 쌓여 되돌리기 어려워진다.
관련 코드를 먼저 읽고 데이터 흐름을 파악한 뒤에만 설계·실행한다.
이 순서를 어기면 "자식 카드가 안 보인다" "알림이 안 온다" 같은
원인 불명 증상이 반복된다.

## 언제 쓰나 (트리거)
- 자동화를 봇/클코에 위임하기 직전
- 기존 코드(broker agent.js, 대시보드 route.ts 등)를 고치기 직전
- 새 기능을 설계하기 직전
- "그냥 이렇게 하면 되지 않나" 추측이 들 때 (이때가 가장 위험)

## 사용법
순서 절대 — 추측으로 ④부터 시작 금지.
① 관련 코드 읽기   — 고칠/부를 함수의 실제 본문을 먼저 본다
② 데이터 흐름 파악  — 입력→처리→출력, 어디서 무엇이 만들어지고 전달되나
③ 설계            — 흐름 위에서 변경점을 정한다 (어느 줄, 어떤 가드)
④ 실행            — 그제서야 코드를 고치거나 명령을 돌린다

부속 규칙:
- 모름은 추측으로 메우지 않는다. 직접 못 보는 정보(어드민 실물 / DB 실시간
  상태 / 공수 견적)는 "이건 모릅니다, 확인하려면 [구체 명령어]" 형식으로.
- 클코 자동 커밋 금지. 명시 지시 시에만 add/commit/push, 매 명령에
  "git status --short까지만" 포함.
- 긴 파일은 head/tail/sed 범위 출력 또는 사진 업로드로 접힘 우회.
