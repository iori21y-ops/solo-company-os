## 목적
세션이 바뀌어도 새 Claude가 맥락을 잃지 않게 한다. 새 창이 헤매는 원인은
(a) 현재 상태를 못 따라옴 (b) 판단 근거를 못 따라옴 둘이라, 이 둘을
선제 차단하는 구조로 쓴다. 정보를 압축하면 새 Claude가 딴소리하므로 자세히 쓴다.

## 언제 쓰나 (트리거)
- 경년님이 "인수인계 작성해줘"라고 명시 지시했을 때만. (자동 생성 금지)
- 한 세션=한 파트가 끝나 다음 파트로 넘어갈 때
- 대화가 길어져 성능 저하 우려로 새 세션을 권할 때

## 사용법
맨 위에 "🔑 새 창 빠른 복귀" 핀 블록 고정 → 그 아래 7단계 본문.

[🔑 새 창 빠른 복귀 핀 — 항상 맨 위 고정]
- 지금 위치: [이번 세션 어디까지 했는지]
- 핵심 왜: [중요 판단 1~2개 + 이유]

[7단계 본문]
1. "이어가는 중"이라 명시 (새 작업 아니라 연속임을 신호)
2. 마스터플랜/큰 그림부터 (왜 이걸 하는가)
3. 관련 시스템 뼈대 / 이미 만든 자산 나열
4. 현재 미션 작업 흐름과 1차 결과
5. 판단 바뀐 부분이 있으면 "왜 바뀌었는지" 추론 과정 노출
6. 비즈니스 우선순위 / 범위 명시
7. 이번 세션 첫 지시 3~5개로 좁힘 + 제약 명시
   (예: "코드 수정 금지, 보고서만")
마지막에 환경 정보 (경로 / 키 이름 / 모델 분담).

세션 운영: 한 세션=한 파트. 핸드오프 메모는 200자 이내, 상세는 DECISIONS.md에.
단 인수인계서 본문(핀+7단계)은 압축 금지하고 자세히 쓴다 — 짧은 메모와 역할이 다름.
경년님은 화면 복붙 방식이라 git 명령어 등 불필요, 대화 내용만으로 채움.
