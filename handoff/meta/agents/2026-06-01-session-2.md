## 5. 판단(추론 노출)
저장입구는 봇이 뭘 뱉든 "마지막 메시지"를 저장하므로 ESCALATE 메시지로도 배관 검증 성립. 그래서 이번 세션은 배관·견고화까지로 닫음. content봇이 실제 초안 대신 "검증된 데이터 없음→데이터팀장 검증요청→사장님 결정요청(ESCALATE)"을 올린 건 SKILL.md 원칙(verdict green일 때만 초안)을 따른 정상동작. 즉 다음 미싱피스는 "content봇 정문을 여는 데이터 검증 흐름"이다. ESCALATE를 WP에 발행하는 건 무의미하므로 cycle 1바퀴 실발행은 데이터흐름 깐 뒤로 미룸.

## 6. 우선순위/범위
1순위 = 데이터 검증 흐름 설계: data봇이 차종 데이터 정합성 verdict(green)를 어디에 어떤 형식으로 남기고, content봇이 그걸 어떻게 읽어 초안 해금하는가. 추측 금지 — content/SKILL.md(124줄)의 "verdict green" 규칙 실물과 data 에이전트 산출물 구조부터 읽기.
범위: MVP. 검증흐름 1개 차종으로 통과시켜 content봇이 실제 초안 1편 산출 → 저장→승인→wp_publish.py 발행까지 cycle 1바퀴.

## 7. 첫 지시(순서대로)
1. content/SKILL.md에서 "정합성 verdict/green/검증" 관련 규칙 원문 확인 — content봇이 정확히 무엇을 보고 초안을 허용/거부하는지. grep -niE "verdict|green|정합성|검증" ~/projects/_meta/agents/content/SKILL.md
2. data 에이전트가 verdict를 남기는 산출물 구조 확인 — ls -R ~/projects/_meta/agents/data/ + SKILL.md. 검증결과를 어디 어떤 파일/형식으로 쓰나.
3. 둘 사이 연결 설계(추측 금지, 코드·구조 읽고): data verdict 산출물 → content봇이 읽는 경로/형식 합의메모.
4. 합의 후 구현(str_replace·신규생성만) → 1개 차종으로 검증흐름 통과 → content봇 실제 초안 산출 확인.
5. 초안 나오면 cycle 1바퀴: 저장→승인→wp_publish.py 발행. 합의메모+cycle/playbook 박제.
