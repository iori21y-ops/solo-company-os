## 5. 판단(추론 노출)
정합성 체크를 "가격/트림 점검" 하나로 좁히려다 두 번 막혔다. 본질은 데이터 유형마다 다르고(5유형), 각 유형의 점검 방식이 _config audit-rules 5종으로 이미 분리돼 있다. 그래서 verdict는 단일 형식이되 data_type+audit_kind 필드로 "어떤 유형이 어떤 방식으로 검증됐나"를 표기한다. content는 자기가 쓸 유형의 green만 본다(가격글=quantitative green, 이미지=visual green, 유형 다르면 상호 보증 안 함). _config 5종 실내용(무엇이 green인가)을 다 채우는 건 큰 작업이라, 점검 실물이 이미 있는 visual/referential(wf_vi) 1유형으로 배관부터 돌린다. source_cycle은 MVP에서 null(학습루프 역추적용, 해금 판정과 무관, cycle 파트에서 채움).

## 6. 우선순위/범위
1순위 = content봇이 data/outputs/approved/의 verdict 파일을 읽어 해당 차종(subject 필드) verdict:green을 확인하고 초안을 허용/거부하는 로직. 추측 금지 — content/SKILL.md에서 content봇이 초안 전 무엇을 읽는지(103줄 data approved 전역 스카우팅), agent.js에서 봇 프롬프트가 어떻게 구성되는지 코드부터 읽기.
범위: MVP. visual/referential 1유형·아반떼 1차종. verdict green 1건 만들어 content봇이 그걸 읽고 초안 1편 산출되는지 검증.

## 7. 첫 지시(순서대로)
1. 합의메모 재확인: cat ~/projects/_meta/agents/governance/합의메모-verdict형식.md — 확정된 스키마·해금조건 숙지.
2. content봇이 초안 전 verdict를 어떻게 읽는지 현재 코드 확인: agent.js에서 content봇 프롬프트/SKILL 주입 방식 + content/SKILL.md 103·106줄 맥락. green 판정이 봇 프롬프트로 처리되는지(SKILL 지시) vs 코드로 처리되는지 파악.
3. 둘 중 어디서 green을 검사할지 설계(추측 금지, 코드 읽고): SKILL 지시만으로 봇이 verdict 파일 읽고 판단 가능한지 / agent.js에 verdict 스캔 로직이 필요한지.
4. 합의 후 구현(str_replace·신규생성만): 아반떼 verdict green 1건을 data/outputs/approved/에 수동 생성 → content봇이 읽고 초안 허용하는지 디스코드 실테스트.
5. 통과하면 cycle 1바퀴: 초안→저장→승인→wp_publish.py 발행. 합의메모+cycle/playbook 박제.
