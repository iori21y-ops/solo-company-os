## 4. 이번 세션 작업 흐름·결과
①코드구조 파악: systemPrompt=main.js가 시작시 SKILL 1회읽어 고정/--dangerously-skip-permissions/--no-session-persistence(매턴 새세션) ②설계: 방법A(SKILL지시)vs B(코드스캔) 중 B확정 — no-session-persistence라 SKILL지시 비결정론, 코드가 결정론적 주입 ③게이트문구 동봉 결정(데이터만 주면 비결정론 절반 남음) ④str_replace로 314줄 userPrompt 교체 ⑤문법통과 ⑥아반떼 verdict 생성 ⑦재기동 ⑧디스코드 실증: 카니발=차단+ESCALATE✅ / 아반떼=green통과→CLI도달✅ / 안녕=즉답✅ / 아반떼 가격글=visual green이나 quantitative미검증이라 봇이 정확히 거부(유형분리 작동)✅.

## 5. 판단(추론 노출)
게이트는 양방향 완전 작동(로그 9108~9268줄 실증). 그러나 아반떼 스펙카드(도구사용 작업)에서 5분 타임아웃. 원인을 "복잡작업이라 오래걸림"으로 단정 안 함 — 두 경우 미구분: (가)정말 진행중인데 시간부족 (나)도구호출이 행(hang). 단서: CLI단독 단순질문 3.2초, broker경유 단순질문도 즉답, 그런데 도구사용 복잡작업만 300초 꽉채우고 죽음. broker 자체는 정상(단순호출 통과)이라 broker버그 아님. macOS에 timeout 명령 없어 CLI단독 복잡작업 재현은 못함(gtimeout 또는 time claude -p 직접). CLAUDE_MAX_TURNS 설정값 미확인 — multi-turn 도구반복 호출 중 행 가능성.

## 6. 우선순위/범위
1순위 = 도구사용 복잡작업 타임아웃 해결. 추측 금지 — ①agent.js의 claude 호출 옵션 전체(--max-turns, --allowedTools 등 도구관련) 읽기 ②.env의 CLAUDE_MAX_TURNS 등 ③broker.log에서 타임아웃건 직전 마지막 stdout/stderr가 무엇이었나(어느 단계서 멈췄나) ④CLI단독으로 같은 복잡작업 직접 재현(time claude -p, real시간 측정)해서 (가)/(나) 가르기.
주의: 단순히 타임아웃 숫자만 늘리는 건 (나)면 헛수고. 원인 가른 뒤 해결안 1개.
범위: 타임아웃 해결까지. 풀리면 그 세션 또는 다음에 cycle 1바퀴(아반떼 visual 에셋 산출→저장→승인→발행).

## 7. 첫 지시(순서대로)
1. 합의메모 숙지: cat governance/합의메모-content봇-verdict읽기.md + 합의메모-verdict형식.md.
2. claude 호출 옵션·타임아웃 관련 코드 읽기: grep -nE "spawn|exec|claude|--|MAX_TURNS|allowedTools|timeout|TIMEOUT" agent.js + grep -E "TIMEOUT|MAX_TURNS|MODEL" .env (값만, 토큰노출금지).
3. broker.log에서 아반떼 스펙카드 타임아웃건(약 9256~9260줄 부근) 직전 마지막 출력 확인 — 어느 단계서 멈췄나.
4. CLI단독 복잡작업 재현으로 (가)진행중/(나)행 가르기: time claude -p "아반떼 이미지 스펙 카드 설계해서 텍스트로 답해줘" (gtimeout 있으면 감쌈). real시간이 분기 결정.
5. 원인 확정 후 해결안 1개 합의 → str_replace·신규생성만 구현 → 디스코드 재테스트(아반떼 스펙카드가 거부없이 산출되면 게이트열림 최종확정).
