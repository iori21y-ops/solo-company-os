# 데이터팀 업무 레지스트리 — 주기/스팟 분류 + 실행 스펙

> 지시: "데이터 할일을 주기/스팟으로 구분해 파악 → 데이터에 직접 전달해 수행 → 결과보고를 관리팀장에게."
> 원칙(advisor 점검 반영): **새 메커니즘 금지.** 보고루프·분류틀·thresholds는 이미 존재 → 재사용.
> 이 문서가 채우는 공백은 단 두 가지 — ① 각 항목의 **실행 스펙(트리거/입력/산출/합격기준)**, ② 주기 항목의 **배치 보고 cadence·포맷**.

## 보고 루프 (재확인 — 기존 그대로, 신규 아님)
data/SKILL.md에 이미 정의된 흐름을 그대로 쓴다:
1. data가 항목 수행 → 산출물을 `data/outputs/pending/`에 저장
2. data가 `[APPROVAL:관리팀장] 제목 | 요약 | /절대경로` 결재 카드로 관리팀장에 보고
3. 관리팀장 1차 검토 → **green이면 경량 승인**(approved/ 이동) / **이슈 있으면 대표 결재** 또는 수정 [CALL:데이터]
4. 사이트 표시 자산 이슈·풀 범위 변경·스팟→주기 승격 → **대표 결재**

**주기 항목 배치 보고(이 문서가 채우는 공백):**
- 일상 예약 수집은 결재 없음(auto). **이상 無 = 조용히 / 이상 有 = 즉시 알림 보고.**
- 정합성·신선도 verdict는 **주 1회 배치**로 모아 1장에 묶어 보고(trigger_type=schedule → 관리팀장 배치 검토). green-all이면 경량 승인.

## Part D. 결정 완료 (2026-06-02, 대표 지시 "자동으로 해")
**주기 A-1~A-3 자동화 ON 확정.** 단, n8n이 아니라 **launchd(맥미니) 스케줄러**로 구현 — 전역 안전게이트(n8n 활성화·서비스 재시작) 미저촉, 산출은 read-only 감사 .md(DB 미수정·표시자산 미변경).
- 구현: `~/projects/_meta/scripts/data-recurring-trigger.sh {a1|a2|a3}` → 데이터 채널 역할멘션 POST(health-check·[CALL:] 검증 패턴 재사용) → broker가 데이터 봇 가동.
- 스케줄(launchctl 등록 완료):
  - `com.rentailor.data-a3-crawl` — **매일 06:00** (크롤 후, 이상 시만 보고)
  - `com.rentailor.data-a1-supply` — **매주 월 06:15**
  - `com.rentailor.data-a2-freshness` — **매주 월 06:30**
- A-4(변동 감지): 순수 cron 불가 → A-3 일일 트리거에 가벼운 확인으로 동봉, 변동 시 스팟 전환 제안만.
- 검토 루프: 데이터가 [APPROVAL:관리팀장] 카드 발행 시 멘션으로 관리팀장 자동 기상 → 1차 검토(이벤트 구동, 별도 cron 불필요).
- 끄기: `launchctl bootout gui/$(id -u)/com.rentailor.data-{a1-supply|a2-freshness|a3-crawl}`
- n8n 상시화(별도 안건)는 여전히 보류 — 이 launchd 방식은 n8n 없이 동작.
