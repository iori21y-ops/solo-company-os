## 증거 (실측)

1. **launchd 설정** (`~/Library/LaunchAgents/com.cadam.price-crawler.plist`)
   - `StartCalendarInterval`: **매월 1일 02:00** 에만 실행
   - `KeepAlive=false`, `RunAtLoad=false` → 죽으면 안 살림, 부팅 시 안 띄움
   - 실행 대상: `run_monthly.sh` (파일명부터 "monthly")

2. **run_monthly.sh 동작 흐름**
   - 서버 기동(`--serve --port 9722`) → 헬스체크 → `POST /crawl {crawl_all:true}` 1회 → **서버 kill 후 종료**
   - 즉, 크롤링 한 번 돌리려고 잠깐 서버를 띄웠다가 끝나면 스스로 닫음

3. **crawler.log 실측**
   - 마지막 실행: **2026-06-01 02:00 ~ 02:08**, 정상 완료
   - 현대·기아 전 차종 수집 + 변경사항 기록 + generate_pricing 정상 → `POST /crawl 200` → 서버 종료 완료
   - **6월치 가격 수집은 이미 정상적으로 끝났습니다.**

4. **launchctl 상태**: `- 0 com.cadam.price-crawler` → 마지막 종료코드 0(정상), 현재 미실행(정상)

## n8n 연동 검증 (오탐 vs 실제장애 가름)

"n8n이 평소에도 9722를 호출하는 구조"라는 데이터팀 설명을 검증했습니다 → **사실 아님**.
- 활성 워크플로우(GWP-1, `active:true`, 매일 23시 스케줄)는 **9721**(`host.docker.internal:9721/crawl`)을 호출 — 이건 현재 정상 가동 중.
- 워크플로우 파일에서 잡힌 "9722"는 전부 UUID 안의 우연한 문자열(`versionId: ...9722...`)이고, **포트 9722를 호출하는 활성 워크플로우는 0개**.
- 9722는 오직 `run_monthly.sh`가 월 1회 자체 curl로만 호출 → 외부 상시 의존성 없음.

→ **A안으로 알림을 꺼도 묻히는 실제 장애가 없습니다.** (9721과 9722 혼동이 그간 오진의 또 다른 원인)
