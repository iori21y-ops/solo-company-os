# 데이터팀이 해야 할 일 — 구현 시스템 역산 분석

> 지시: "현재 구현된 모든 업무를 분석해서 데이터가 해야할일이 뭔지 파악."
> 방법: 실제 구현된 시스템(Supabase 22테이블·크롤러·n8n·WF-VI 이미지)을 까서,
> 데이터팀 SKILL.md 경계규칙(쌍2·쌍4)으로 걸러 data 소유 항목만 추출.

## 한 줄 결론
데이터팀의 일은 **두 갈래**다 — ① 정의돼 있으나 거의 안 돌아간 **상시 책임(정합성·신선도 verdict)을 실제로 켜는 것**, ② 추적 중이나 착수 전인 **WF-VI 이미지 백로그(002/003/004/005)를 처리하는 것**. 후자가 지금 당장 잡을 수 있는 실작업이다.

## 0. 경계 필터 결과 — data가 아닌 것 (먼저 제외)
SKILL.md 판정선 **"실행=ops, 결과검증=data"** 적용:
- ISS-001(n8n insights 저장)·003(Gemini 429)·004(폴백 버그)·005(curated 적체)·006(Pexels 키)·008(토픽 정체)·010/011(infra) → **전부 ops/n8n 실행 이슈. data 아님.**
- `consultations`(상담 KPI 테이블) → 고객·web 산출물. **data 소유 아님.** (헷갈리기 쉬워 명시 제외)
- ※ ISSUES.md는 2026-04-30 갱신본으로 한 달 묵음 → 아래는 stale 인용 대신 가능한 한 라이브 점검 기반.

## 1. 상시 책임 (recurring) — "정의됐으나 미가동" 상태
현재 data outputs는 approved 1건(avante verdict)뿐, pending/rejected 비었고 wiki 사이클 0건.
즉 역할은 박제됐으나 **정기 보고·정합성 verdict가 한 번도 안 돌았다.** 이걸 켜는 게 절반.

| 영역 | 대상 테이블/자산 | data가 켜야 할 일 |
|------|----------------|------------------|
| 차량 공급 데이터 정합성 | vehicles, pricing, vehicle_trims, trim_options, vehicle_msrp, vehicle_used_prices, finance_rates | 정기 정합성 verdict(green/이슈) 발행 → 다운스트림은 green 구역만 소비 |
| 사이트 표시 데이터 신선도 | fuel_prices, ev_chargers, gas_stations, insurance_stats, repair_shops | freshness 임계 점검 + 갱신 알림 |
| 수집 결과 검증 | info_articles (크롤러 산출) | 일일 수집 건수·이상치 탐지·보고 (실행은 ops, 탐지·보고가 data) |
| 외부 시장 데이터 | 환율·보조금·정책 | 변동 감지 시 스팟 수집 |
