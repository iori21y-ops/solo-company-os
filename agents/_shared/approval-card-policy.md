# 결재 카드 표준

## 원칙

> **2026-06-03 변경:** 인터랙티브 버튼 카드를 폐지하고 **평문 알림 + 결재물 링크 + 텍스트 명령 결정** 방식으로 전환했다. (버튼은 게이트웨이 재시작 시 "expired"로 죽고 컨텍스트·턴 비용이 컸다.)

- 모든 결재 요청은 Discord 결재 알림으로 올린다.
- 결재 알림은 **평문 메시지 + 결재물 링크** 형식이다. 버튼·모달은 쓰지 않는다.
- 결재 결정은 **사장님이 결재 알림에 답글**로 한다:
  - `승인` → 브로커가 ref 마커의 파일을 `outputs/pending`에서 `approved`로 이동하고 원채널 담당 에이전트에 통보한다.
  - `수정 [요청]` → 파일은 `pending`에 유지하고 원채널 담당 에이전트에 수정 요청을 통보한다.
  - `반려 [사유]` → 브로커가 ref 마커의 파일을 `outputs/pending`에서 `rejected`로 이동하고 원채널 담당 에이전트에 통보한다.
  - 예전 수동 형식 `승인 <에이전트> <파일명>` / `반려 <에이전트> <파일명>`은 호환용으로만 남긴다.
- 결재 대상 결과물이 있으면 반드시 알림에서 확인 가능해야 한다. 로컬 vault 파일(`--file`)은 자동으로 모바일 뷰어 링크로 변환되고, 웹/문서 결과물은 `--link`로 표시한다.
- Discord 모바일에서 탭 가능하게 결재물 URL은 알림 본문에 `<https://...>` / `<http://...>` 형식으로 넣는다. (`send-approval-card.mjs`가 자동 처리)
- 결과물이 있는데 링크(또는 첨부)도 없는 결재 요청은 유효한 결재로 보지 않는다.
- 결재 알림은 원 업무 채널/스레드에 올린다. `#대표이사실`에 결재를 모으지 않는다. 사장님은 해당 결재 알림에 답글로 `승인`/`수정`/`반려`를 입력한다.
- 결재물 모바일 열람: **md-viewer** `http://macmini.tail466d13.ts.net:9724/` (Tailscale 망 전용, 로그인 없음). 목록 페이지에서 전 에이전트 pending을 한눈에 본다.
- 채팅 공지만으로는 적용된 것으로 보지 않는다. 각 에이전트 지침과 결재 도구에 이 규칙이 있어야 한다.
- **단계 분리 (한 호출에 묶지 말 것)**: 한 번의 호출에 `검토·판단`과 그 후속 동작(결재 카드 상신·`[ESCALATE]`)을 함께 하지 않는다. 검토·판단 결과는 채널에 직접 게시하고(보고 직접화), 후속 동작(카드 상신·에스컬레이션)은 별도 호출로 수행한다. 한 호출에 묶으면 단계·시간 한도(max-turns·timeout)에 걸려 후속 동작 전에 잘린다. (이 규칙의 항상-로드 사본은 `admin/SKILL.md §3`에 있다.)
- **게이트 조건 우선:** 결재물에 `verdict_gate: yellow`, `pending-correction`, `hold`, `확정 전 발행 불가` 같은 차단 조건이 있거나 참조한 approved verdict가 없으면 승인하지 않는다. 먼저 게이트 해제 산출물을 만들고 재상신한다.
- **최종보고 기준:** 승인 완료와 외부 발행 완료를 섞지 않는다. 최종보고에는 `데이터 승인 / 콘텐츠 승인 / 마케팅 발행 / 외부 게시 여부`를 분리해 적는다. 외부 게시가 없으면 `소스 승인 완료, 발행 대기`라고 보고한다.
- **외부 게시 최종 결재:** 마케팅 발행물이 approved/로 이동한 뒤에도 Instagram·블로그·SNS 등 외부 채널에 실제 게시하기 전에는 별도 최종 게시 결재를 올린다. 이 단계는 `마케팅 발행물 승인 완료 / 외부 게시 최종 결재 대기`로 보고하며, 최종 게시 결재 없이 외부 게시를 완료 처리하지 않는다.

## 답글 결재 + 라운드트립 (2026-06-03)

- **결재 결정은 결재 알림에 답글로 한다:** `승인`, `수정 [요청]`, `반려 [사유]`. (브로커가 순수 코드로 처리 — LLM 미호출, 컨텍스트 0)
- 카드에는 `-# [ref] agent=… | origin=… | path=…` 마커가 박혀 있어, 답글 결정 시 ① 파일 결재면 승인=`pending→approved`, 반려=`pending→rejected`, 수정=`pending 유지` ② **원채널(origin)의 담당 에이전트에게 역할멘션으로 통보**된다.
- **담당 에이전트 계약(필수):** 내가 올린 결재가 승인되어 원채널에 `<@&내역할> ✅ 승인: "제목" — 다음 단계 진행하세요` 통보가 오면, **별도 확인코드 없이** 상신 시 밝힌 다음 단계를 바로 진행한다. `📝 수정 요청`이면 요청을 반영해 재상신하고, `❌ 반려`면 사유를 반영해 보류/폐기한다. (이 round-trip 승인이 곧 대표님의 공식 go-signal이다 — DB·코드 등 변이 집행의 "대표승인 후" 요건을 충족한다.)
- 에스컬레이션(`[ESCALATE]` 결정 요청)은 필요한 경우 대표 멘션을 포함하되, 결재가 필요한 결과물은 `[APPROVAL:대표님]`으로 직접 상신한다.

## 결재자

- 모든 에이전트의 결재 기본 결재자는 `대표님`이다.
- 실무자 `마케팅`, `컨텐츠`/`콘텐츠`, `데이터`도 대표님에게 직접 결재 카드를 올린다. 기획팀장·관리팀장을 결재 경유지로 두지 않는다.
- 관리팀장 검토가 필요한 업무라도 결재 라인은 대표님 직접 상신을 유지한다. 관리팀장은 사전 의견이나 리스크 검토만 제공하고 결재자를 대체하지 않는다.
- 채널 override 금지: `#마케팅`(`1510444722433429514`), `#컨텐츠`(`1510446872295379065`), `#데이터`(`1510447003107070155`)에서도 `[APPROVAL:대표님]`을 직접 사용한다.
- 결재 알림 발신자는 원 요청 에이전트와 일치해야 한다. 예: 마케팅 산출물은 마케팅 봇이 `[APPROVAL:대표님]`으로 직접 상신하며, 기획팀장이나 관리팀장이 대행 발송하지 않는다.

## Claude 브로커 프로토콜

Claude 브로커 소속 에이전트는 결재 요청을 일반 텍스트로 끝내지 말고, 응답 마지막에 아래 한 줄을 붙인다.

```text
[APPROVAL:대표님] 제목 | 요청 요약 | /절대/경로/파일명 또는 결과물 링크
```

모든 에이전트가 대표 결재가 필요한 건을 아래 형식으로 직접 상신한다.

```text
[APPROVAL:대표님] 제목 | 요청 요약 | /절대/경로/파일명 또는 결과물 링크
```

예시:

```text
[APPROVAL:대표님] 데이터 검증 리포트 결재 요청 | 아반떼 가격 데이터 정합성 리포트 검토 요청 | /Users/kim/projects/_meta/agents/data/outputs/pending/verdict-avante-20260602.md
```

결재 대상 결과물이 없는 순수 결정 요청이면 세 번째 칸은 생략할 수 있다. 결과물이 있으면 생략하지 않는다.

```text
[APPROVAL:대표님] 발행 후보 승인 요청 | 이번 주 네이버 블로그 후보 3건 중 1건 선택 요청
```

브로커는 이 프로토콜을 감지하면 `tools/send-approval-card.mjs`를 호출해 원 업무 채널/스레드에 **평문 결재 알림**(상신 명의·결재자·결재물 링크·답글 결정 안내)을 게시한다. 버튼은 생성하지 않는다. `[APPROVAL]` 세 번째 칸의 vault 파일 경로는 자동으로 md-viewer 링크로 변환되고, `승인`/`수정`/`반려` 답글 안내가 함께 붙는다.

## 반복 업무 / 엔카 카드뉴스 기준

- 반복 업무로 승격되는 작업은 2회차 전에 SOP를 남긴다: source of truth, verdict 요건, 저장 경로, 결재자, 최종보고 포맷, 중복 호출 방지.
- 엔카 카드뉴스는 데이터팀의 단일 authoritative verdict 1개를 기준으로 한다. JSON/DB 충돌이 있으면 콘텐츠 제작·마케팅 발행을 보류하고, 충돌 원인과 단일 재계산 기준을 먼저 결재한다.
- 콘텐츠 산출 표준 경로는 `~/projects/_meta/agents/content/outputs/*`다. 신규 산출물에 `컨텐츠/` 한글 중복 경로를 쓰지 않는다.
- 데이터 verdict는 표본 부족, 이상치, 단독 인용 금지 수치, 금지 표현을 명시해야 하며 콘텐츠·마케팅은 이를 그대로 보존한다.

## OpenClaw 도구

OpenClaw/Codex 쪽은 다음 스크립트를 사용한다.

```bash
node /Users/kim/.openclaw/workspace/tools/send-approval-card.mjs \
  --target channel:1510181005330350100 \
  --approver 대표님 \
  --title "결재 요청 제목" \
  --summary "요청 요약" \
  --file "/절대/경로/파일명"
```

파일 대신 링크를 보여줄 때:

```bash
node /Users/kim/.openclaw/workspace/tools/send-approval-card.mjs \
  --target channel:1510181005330350100 \
  --approver 대표님 \
  --title "결재 요청 제목" \
  --summary "요청 요약" \
  --link "https://example.com/result"
```

`--approver`는 `대표님`, `기획팀장`, `관리팀장`, Discord 사용자 ID를 지원한다.
