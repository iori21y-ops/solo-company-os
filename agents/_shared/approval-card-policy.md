# 결재 카드 표준

## 원칙

> **2026-06-03 변경:** 인터랙티브 버튼 카드를 폐지하고 **평문 알림 + 결재물 링크 + 텍스트 명령 결정** 방식으로 전환했다. (버튼은 게이트웨이 재시작 시 "expired"로 죽고 컨텍스트·턴 비용이 컸다.)

- 모든 결재 요청은 Discord 결재 알림으로 올린다.
- 결재 알림은 **평문 메시지 + 결재물 링크** 형식이다. 버튼·모달은 쓰지 않는다.
- 결재 결정은 **사장님이 텍스트 명령**으로 한다:
  - `승인 <에이전트> <파일명>` 또는 `반려 <에이전트> <파일명>` → 브로커가 `outputs/pending`에서 `approved`/`rejected`로 파일을 이동한다 (pending 파일에만 동작).
  - 조건부 승인·수정 요청·반려 사유 등 의견은 같은 채널에 한 줄로 남긴다.
- 결재 대상 결과물이 있으면 반드시 알림에서 확인 가능해야 한다. 로컬 vault 파일(`--file`)은 자동으로 모바일 뷰어 링크로 변환되고, 웹/문서 결과물은 `--link`로 표시한다.
- Discord 모바일에서 탭 가능하게 결재물 URL은 알림 본문에 `<https://...>` / `<http://...>` 형식으로 넣는다. (`send-approval-card.mjs`가 자동 처리)
- 결과물이 있는데 링크(또는 첨부)도 없는 결재 요청은 유효한 결재로 보지 않는다.
- **결재 알림은 전용 `#결재함` 채널(`1510170922005565470`)로 중앙 집결한다.** 작업 채널/스레드에 흩어지지 않는다. (`send-approval-card.mjs`가 `--target`과 무관하게 결재함으로 강제 라우팅) 사장님은 결재함만 보고 `승인/반려`를 입력한다.
- 결재물 모바일 열람: **md-viewer** `http://macmini.tail466d13.ts.net:9724/` (Tailscale 망 전용, 로그인 없음). 목록 페이지에서 전 에이전트 pending을 한눈에 본다.
- 채팅 공지만으로는 적용된 것으로 보지 않는다. 각 에이전트 지침과 결재 도구에 이 규칙이 있어야 한다.
- **단계 분리 (한 호출에 묶지 말 것)**: 한 번의 호출에 `검토·판단`과 그 후속 동작(결재 카드 상신·`[ESCALATE]`)을 함께 하지 않는다. 검토·판단 결과는 채널에 직접 게시하고(보고 직접화), 후속 동작(카드 상신·에스컬레이션)은 별도 호출로 수행한다. 한 호출에 묶으면 단계·시간 한도(max-turns·timeout)에 걸려 후속 동작 전에 잘린다. (이 규칙의 항상-로드 사본은 `admin/SKILL.md §3`에 있다.)

## 결재함 답글 결재 + 라운드트립 (2026-06-03)

- **결재 결정은 결재함에서 카드에 답글로 한다:** `승인` 또는 `반려 [사유]`. (관리팀장 봇이 순수 코드로 처리 — LLM 미호출, 컨텍스트 0)
- 카드에는 `-# [ref] agent=… | origin=… | path=…` 마커가 박혀 있어, 답글 승인 시 ① 파일 결재면 `pending→approved/rejected` 이동 ② **원채널(origin)의 담당 에이전트에게 역할멘션으로 통보**된다.
- **담당 에이전트 계약(필수):** 내가 올린 결재가 승인되어 원채널에 `<@&내역할> ✅ 승인되었습니다: "제목" — 다음 단계 진행하세요` 통보가 오면, **별도 확인코드 없이** 상신 시 밝힌 다음 단계를 바로 진행한다. `❌ 반려`면 사유를 반영해 보류/수정한다. (이 round-trip 승인이 곧 대표/관리팀장의 공식 go-signal이다 — DB·코드 등 변이 집행의 "대표승인 후" 요건을 충족한다.)
- 에스컬레이션(`[ESCALATE]` 결정 요청)도 결재함으로 모이며, 대표 멘션이 카드 안에 포함된다.

## 결재자

- 실무자 `마케팅`, `컨텐츠`, `데이터`가 관리팀장에게 결재를 올릴 때: 결재자 `관리팀장`
- 기획팀장 또는 관리팀장이 대표님 결정을 받아야 할 때: 결재자 `대표님`
- 실무자 `마케팅`, `컨텐츠`, `데이터`는 대표님에게 직접 결재 카드를 올리지 않는다. 대표 결재가 필요한 사안도 먼저 `관리팀장`에게 결재보고하고, 기획팀장 또는 관리팀장이 검토 의견을 붙여 `대표님`에게 상신한다.
- 기획팀장/OpenClaw 산출물 결재: 기본 결재자 `대표님`, 실무 검토가 먼저 필요한 경우 `관리팀장`
- 채널 override: `#마케팅`(`1510444722433429514`), `#컨텐츠`(`1510446872295379065`), `#데이터`(`1510447003107070155`)에서는 대표결재 사안도 관리팀장이 진행한다. 기획팀장은 이 스레드들의 대표결재 카드를 직접 상신하지 않는다.

## Claude 브로커 프로토콜

Claude 브로커 소속 에이전트는 결재 요청을 일반 텍스트로 끝내지 말고, 응답 마지막에 아래 한 줄을 붙인다.

```text
[APPROVAL:관리팀장] 제목 | 요청 요약 | /절대/경로/파일명 또는 결과물 링크
```

기획팀장 또는 관리팀장만 대표 결재가 필요한 건을 아래 형식으로 상신할 수 있다.

```text
[APPROVAL:대표님] 제목 | 요청 요약 | /절대/경로/파일명 또는 결과물 링크
```

예시:

```text
[APPROVAL:관리팀장] 데이터 검증 리포트 결재 요청 | 아반떼 가격 데이터 정합성 리포트 검토 요청 | /Users/kim/projects/_meta/agents/data/outputs/pending/verdict-avante-20260602.md
```

결재 대상 결과물이 없는 순수 결정 요청이면 세 번째 칸은 생략할 수 있다. 결과물이 있으면 생략하지 않는다.

```text
[APPROVAL:관리팀장] 발행 후보 승인 요청 | 이번 주 네이버 블로그 후보 3건 중 1건 선택 요청
```

브로커는 이 프로토콜을 감지하면 `tools/send-approval-card.mjs`를 호출해 동일 채널에 **평문 결재 알림**(상신 명의·결재자·결재물 링크·`승인/반려` 명령 안내)을 게시한다. 버튼은 생성하지 않는다. `[APPROVAL]` 세 번째 칸의 vault 파일 경로는 자동으로 md-viewer 링크로 변환되고, `승인/반려` 명령 안내가 함께 붙는다.

## OpenClaw 도구

OpenClaw/Codex 쪽은 다음 스크립트를 사용한다.

```bash
node /Users/kim/.openclaw/workspace/tools/send-approval-card.mjs \
  --target channel:1510181005330350100 \
  --approver 관리팀장 \
  --title "결재 요청 제목" \
  --summary "요청 요약" \
  --file "/절대/경로/파일명"
```

파일 대신 링크를 보여줄 때:

```bash
node /Users/kim/.openclaw/workspace/tools/send-approval-card.mjs \
  --target channel:1510181005330350100 \
  --approver 관리팀장 \
  --title "결재 요청 제목" \
  --summary "요청 요약" \
  --link "https://example.com/result"
```

`--approver`는 `대표님`, `기획팀장`, `관리팀장`, Discord 사용자 ID를 지원한다.
