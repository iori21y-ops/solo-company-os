## 호출 방법
### 0. 채널 라우팅 원칙
- `#대표이사실`에서 대표님이 다른 에이전트 업무 지시를 위임한 경우, 실무 에이전트를 `#대표이사실`에서 직접 호출하지 않는다. 그 에이전트들이 참여한 실제 업무 채널에서 호출해야 한다.
- 기본 라우팅: `#컨텐츠-제작소` channel id `1510181005330350100`
- 에이전트별 스레드: `#컨텐츠` channel id `1510446872295379065`, `#마케팅` channel id `1510444722433429514`, `#데이터` channel id `1510447003107070155`
- `#대표이사실`에는 호출 전문을 반복하지 말고, 어디로 라우팅했는지와 현재 단계만 요약 보고한다.

### A. broker 봇이 다른 에이전트를 호출할 때
- 응답 본문과 별도 줄에 `[CALL:에이전트명] 요청 내용` 작성
- broker는 `[CALL:]`을 해당 에이전트의 디스코드 역할멘션으로 변환해 발송한다
- 자기 자신은 호출하지 않는다
- 한 응답에 여러 에이전트 호출 가능 (각각 별도 줄)
- 직접 처리 가능한 작업은 호출하지 말 것
- 호출 없는 일반 응답에는 [CALL:] 태그를 쓰지 않는다
- 다른 에이전트가 부른 업무에 답하고 더 넘길 호출·결재·대표 결정 요청이 없으면 마지막 줄에 `종료`를 쓴다

### 호출 절약·보고 원칙
- 같은 이슈로 같은 에이전트를 반복 호출하지 않는다. 관리팀장이 맥락을 1회 요약해 필요한 담당자에게만 bounded request로 보낸다.
- 순수 보고, 진행상태 공유, 완료 알림은 [CALL:] 금지. 채널에 직접 보고하고 관리팀장은 채널 로그를 읽어 요약한다.
- 다른 봇이 보낸 `종료`, `응답 없음`, `session limit/429`, `구독 한도`, `Missing Access`, `ChannelNotCached` 같은 종료·오류 알림은 새 업무 지시로 해석하지 않는다.
- 한 호출에는 한 단계만 담는다. `검토→판단→수정지시→결재상신→대표보고`를 한 번에 묶지 않는다.
- 최종보고는 반드시 단계별로 분리한다: `데이터 승인`, `콘텐츠 승인`, `마케팅/외부 발행`, `보류`, `완료`. 외부 발행이 없으면 `발행 대기`라고 명시한다.
- 결재 카드 1건은 산출물 1개만 다룬다. 중복 파일·중복 경로가 있으면 먼저 정리한 뒤 결재한다.
- 게이트 조건(`yellow`, `pending`, `hold`, `확정 전 발행 불가`, approved verdict 누락)이 있으면 승인보다 게이트 해소가 우선이다.

### B. 직접 역할멘션
- 디스코드 역할 멘션 `<@&역할ID>`도 유효하다
- broker 응답 본문 안의 위 표 역할멘션은 실제 ping이 허용된다
- 예시: `<@&1510179969354170481> 이번 달 콘텐츠 기획안 검토 요청`

### C. OpenClaw 봇이 다른 봇 호출할 때
- 디스코드 역할 멘션 `<@&역할ID>` 사용
- `#대표이사실`에서는 실무 에이전트 직접 호출 금지. 확인된 `#컨텐츠-제작소` 또는 에이전트별 스레드 세션으로 라우팅해 호출한다.

## 종료 규칙
- 봇 간 상호 호출에는 답변 횟수 제한을 두지 않는다
- 대신 각 에이전트는 자기 차례의 처리가 끝났고 추가 호출이 없으면 마지막 줄을 `종료`로 닫는다
- `종료`는 해당 응답 체인이 끝났다는 신호다. 새 지시는 대표님 또는 새 역할멘션/호출로 다시 시작한다

## 업무 칸반 기록 규칙 ([TASK])
실질 업무를 시작/완료할 때 ERP 업무 칸반에 카드를 남긴다. broker가 이 태그를
감지해 대시보드(/api/tasks)에 기록한다. (담당자 이름은 broker가 자동 기입)

- 착수 시: 응답 본문과 별도 줄에 `[TASK:진행중] 업무제목 | 출처 | 우선순위`
- 완료 시: 별도 줄에 `[TASK:완료] 업무제목`
  ⚠️ 완료의 제목은 착수 때와 **글자까지 동일**해야 한다. 다르면 카드를 못 찾는다.
- 보류/취소: `[TASK:보류] 제목` / `[TASK:취소] 제목` (주로 관리팀장)
- 산출물 있으면 뒤에 ` | result=경로` (예: `[TASK:완료] 엔카 시세수집 | result=agents/data/outputs/approved/verdict-x.md`)

값: 출처=`스팟`|`주기`|`디스코드`, 우선순위=`높음`|`보통`|`낮음`(생략 시 보통)
원칙: 산출이 있는 업무 단위에만. 연 카드는 반드시 완료/보류/취소로 닫는다. 순수 보고엔 쓰지 않는다.

## 진행 기록 규칙 ([LOG])
업무를 진행하며 "흐름이 바뀐 순간"만 진행 기록으로 남긴다. broker가 이 태그를
감지해 해당 카드의 진행 기록(erp_task_progress_log)에 적재한다. (작성자=broker가 봇 이름 자동 기입)

형식(별도 줄, 파이프 4칸 고정·빈 필드도 자리 유지):
`[LOG:log_type|outcome|related_skill|artifact_url] 본문`
- 예: `[LOG:note|||] 업무 착수, 요건 정리`
- 예: `[LOG:output|success|3-supabase-query|https://.../v1.md] 견적 비교표 작성`

필드값:
- `log_type` = `decision` | `output` | `note` (필수)
- `outcome` = `success` | `fail` | `retry` | `blocked` (없으면 빈칸)
- `related_skill`, `artifact_url` = 없으면 빈칸 (파이프는 유지)

남기는 기준 (★ 빡빡하게 — 과다 금지):
- `decision`: 분기/방향을 실제로 택한 순간에만 1건. 단순 실행은 decision 아님.
- `output`: 산출물(파일·URL·DB행 등)을 실제로 만든 순간에만 1건.
- `note`: 다음 작업에 직접 영향 주는 것만. 그 외에는 남기지 말고 침묵.
- 1사이클당 총 1~3건으로 수렴. 3건 초과면 note부터 버린다.
- 직전 LOG와 의미가 중복이면 출력하지 않는다.
