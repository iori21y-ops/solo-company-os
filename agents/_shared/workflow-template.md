### 7-4. 감시의 위치 + 감시의 감시

| 항목 | 내용 |
|------|------|
| 감시 실행 주체 | n8n 별도 워크플로 / Edge Function / cron |
| 감시가 보는 데이터 | (6번 상태 저장 위치) |
| 감시 자체의 헬스체크 (이중) | **방식 A:** 정기 보고 누락 = 이상 신호 <br> **방식 B (필수 병행):** healthchecks.io 외부 서비스에 감시가 주기적 핑 → 핑 끊기면 healthchecks.io가 사람에게 직접 알림 |
| 마지막 핑 timestamp 저장 위치 | (예: Supabase workflow_health 테이블) — 감시가 핑 보낸 마지막 시각 기록 |
| 정기 보고 누락 시 사람의 행동 | n8n 상태 확인 / Supabase workflow_health 테이블 timestamp 확인 |

**healthchecks.io 보충 설명 (비개발자용):**
무료 외부 서비스. 우리 시스템이 "나 살아있어" 신호를 주기적으로 보냄. 안 보내면 healthchecks.io가 사람에게 이메일/Slack 알림. 우리 시스템이 죽으면 우리 시스템이 알림 못 보내니까 외부에 맡기는 것.

| 노드 | v1 미션 ID | 미션명 | v1 정의서 작성 상태 |
|------|-----------|--------|---------------------|
| 노드 1 | M-XXX | | 작성완료 / 작성중 / 미작성 |
| 노드 2 | | | |

**작성 순서 강제:**
- v1 미션 정의서가 "미작성" 상태인 노드가 있으면 본 워크플로는 "초안" 상태에서 못 벗어남
- 모든 노드의 v1 미션이 "작성완료"여야 워크플로 상태를 "검토중" 이상으로 올릴 수 있음
