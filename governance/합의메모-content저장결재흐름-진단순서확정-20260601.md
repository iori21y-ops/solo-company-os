## 배경
직전 세션(타임아웃 파트) 핸드오프 §3은 "content봇은 초안 텍스트 출력까지만, 저장은 저장입구가"로 적었으나, 파트C 협업결재 합의(§8)와 충돌함이 확인됨. 이 메모는 직전 핸드오프의 틀린 방향을 바로잡고 올바른 순서를 확정한다.

## 확인된 사실 (코드 실측, src/agent.js)
1. 저장·승인·반려·대기 = 전부 사장님 전용 수동 디스코드 명령 (!message.author.bot, line 177 AGENT_BOT_IDS=content/marketing/data).
   - 저장 content {파일명} → 봇 최근 메시지를 pending/에 writeFileSync (line 194)
   - 승인/반려 content {파일명} → renameSync pending→approved/rejected (line 157)
2. ESCALATE: 봇 응답에 [ESCALATE] 포함 → line 521 파싱 → line 262 사장님 DM. line 329/331: 데이터 정합성 verdict 없으면 봇에게 강제 ESCALATE 지시.
3. 관리팀장 자동 결재판 = 없음 (setInterval/watchPending 없음). 결재판 커맨드(line 112)도 사장님 수동.

## 파트C 목표 설계 (§8)
실무봇 산출 → outputs/pending/ → 관리팀장 1차 품질검토 → 정기·저위험=관리팀장 승인 / 스팟·고위험=대표 ESCALATE → approved/.

## 진단 결론
1. 봇이 pending에 저장하는 것은 정상(파트C 설계대로). 직전 세션이 이를 "저장입구 우회=버그"로 본 것은 오판.
2. 진짜 문제 = ①봇이 정기 업무인데 대표에게 직접 ESCALATE ②관리팀장 결재판(정기 승인) 부재.
3. 순서: 받는 쪽(관리팀장 결재판)을 먼저, 보내는 쪽(봇 SKILL)을 나중. 거꾸로 하면 pending에 쌓이기만 하고 통과 안 됨.
